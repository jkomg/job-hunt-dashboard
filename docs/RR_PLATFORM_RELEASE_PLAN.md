# Remote Rebellion Platform Release Plan

This release shifts Job Hunt Dashboard from a single-user/self-hosted tracker into a hosted, multi-tenant job search command center for Remote Rebellion members and staff.

The initial production tenant is `Remote Rebellion`, but the data model should support additional organizations from the start. The UI can stay single-organization until there is a real need to expose tenant switching.

## Goals

- Give job seekers one command center for pipeline, interviews, events, networking, check-ins, follow-ups, and briefing.
- Give Remote Rebellion staff a controlled admin workspace for adding opportunities, flagging follow-ups, and supporting users.
- Make the hosted app the operational source of truth.
- Keep Google Sheets as an integration/export/backup path, not the primary operating system.
- Preserve the low-cost Cloud Run deployment path for early testing.
- Keep the local/self-hosted mode available for users who want a standalone tracker.

## Non-Goals For Initial Release

- No public self-serve signup.
- No paid billing/subscription flow.
- No enterprise SSO/IAP requirement.
- No broad marketplace or multi-organization UI.
- No Circle.so write integration until the core RR workflow is validated.

## Users And Roles

### Job Seeker

- Owns their profile, pipeline, check-ins, interviews, events, contacts, notes, and follow-ups.
- Sees jobs added by RR staff and who/what team added them.
- Updates status, notes, application dates, follow-up dates, resume links, and cover letters.
- Can export their data.

### RR Staff

- Views only assigned users or permitted cohorts.
- Adds jobs directly into a user's pipeline.
- Flags follow-ups or "needs attention" items.
- Adds staff notes with clear visibility:
  - `internal_staff`
  - `shared_with_user`
- Reviews user-visible pipeline progress and support context.

### Admin

- Manages organizations, users, staff, roles, assignments, and integrations.
- Reviews audit logs and sync/export history.
- Can disable accounts and trigger exports/backups.

## Tenant Model

Build multi-tenant from the database/API layer first, even though the first UI release ships with one organization.

Core entities:

- `organizations`
- `users`
- `memberships`
- `staff_assignments`
- `pipeline_entries`
- `daily_logs`
- `contacts`
- `interviews`
- `events`
- `templates`
- `watchlist`
- `staff_notes`
- `audit_log`
- `sync_runs`
- `export_runs`

Required ownership fields:

- User-owned records: `organization_id`, `user_id`
- Staff/admin records: `organization_id`, `created_by_user_id`
- Staff assignments: `organization_id`, `staff_user_id`, `job_seeker_user_id`
- Audit records: `organization_id`, `actor_user_id`, `target_user_id`, `action`, `entity_type`, `entity_id`, `created_at`

Every API query that reads or writes user data must enforce tenant and role scope server-side. Frontend filtering is not access control.

## Source Of Truth

The database is the source of truth.

Remote Rebellion staff should add jobs through the app interface. The app stores those jobs in Turso/libSQL and exposes them immediately to the user.

Google Sheets behavior:

- Import existing RR spreadsheet data during migration.
- Optionally import sheet rows during transition.
- Export/back up app data to Google Sheets on a schedule.
- Push selected status fields back only when explicitly enabled.
- Never let Sheets silently overwrite app-owned user updates without a visible sync conflict.

## Auth Plan

Initial rollout:

- Username/password.
- Invite-only account creation.
- Forced password change on first login.
- Public session-mode deploys require a non-default seeded password secret.
- Admin-created staff accounts.
- Strong `SESSION_SECRET`, rate limiting, CSRF, and session expiration.

Next auth step:

- App-managed Google OAuth for convenience.
- This does not require Google IAP or an external HTTPS load balancer.

Advanced/optional:

- Google IAP remains available for deployments that accept load-balancer cost.

## Privacy And Data Safety

Required before RR beta:

- Staff cannot view all users by default.
- Staff access is assignment/cohort scoped.
- Daily check-ins and notes need explicit visibility.
- Staff notes distinguish internal notes from user-visible notes.
- Every staff action that touches user data is audit logged.
- User export and account disable/delete paths exist.
- Admins can inspect sync/export failures.

Recommended visibility model:

- `private_user`: only the job seeker.
- `shared_with_staff`: job seeker plus assigned staff/admins.
- `internal_staff`: staff/admin only.
- `system`: generated by sync/import/automation.

## Phase 1: Multi-Tenant Foundation

Purpose: make the existing single-user app safe for multiple users.

Deliverables:

- Add `organizations`, `memberships`, and role model.
- Add `organization_id` and `user_id` ownership to user data tables.
- Backfill current data into a default `Remote Rebellion` organization for hosted deployments.
- Enforce tenant/user scoping in every API route.
- Add staff/admin role helpers.
- Add session expiration and cleanup.
- Add invite/admin create-user flow.
- Preserve local single-user mode with one default org/user.

Exit criteria:

- Two users cannot see or mutate each other's data.
- Admin can create a user and force first password change.
- Existing local Docker install still works.

## Phase 2: RR Staff Workspace

Purpose: let RR staff support users inside the app.

Deliverables:

- Staff dashboard with assigned users.
- User support detail view.
- Add job to user's pipeline.
- Flag follow-up or "needs attention".
- Add internal/shared staff notes.
- Filter users by stale follow-up, interviewing, no recent activity, and new RR-added jobs.
- Audit log for staff actions.

Exit criteria:

- Staff can add a job for a user without using Google Sheets.
- User sees the new job in their pipeline and briefing where relevant.
- Admin can review who added/changed a user-facing item.

## Phase 3: RR Data Integration And Backup

Purpose: transition from spreadsheet-assisted workflow to database-first workflow.

Deliverables:

- One-time import from current RR spreadsheet.
- Optional transitional import for rows still added to Sheets.
- Scheduled export/back up from DB to Google Sheets.
- Sync status page for admins.
- Conflict report for any two-way fields.
- Configurable field map for RR sheet columns.

Default policy:

- DB wins for app-owned fields.
- Sheets can create new opportunities only during the transition period or when explicitly enabled.
- Sheets export is for continuity, reporting, and recovery.

Exit criteria:

- RR can stop entering new jobs in Sheets for pilot users.
- A daily Google Sheets backup can be regenerated from DB state.
- Sync/export failures are visible to admins.

## Phase 4: Production Hardening

Purpose: make the hosted beta reliable enough for real RR users.

Deliverables:

- App-managed Google OAuth.
- Password reset/invite emails.
- Data export/delete flow.
- Automated backups and restore drill.
- Smoke tests for login, create job, update pipeline, staff add job, and sync/export.
- Error tracking and admin-visible operational health.
- Security checklist for hosted RR deployment.

Exit criteria:

- A non-developer RR admin can onboard a user.
- A failed sync/export is visible and actionable.
- Restore has been tested from a recent backup.

## Phase 5: Product Polish

Purpose: make the app feel like a complete command center.

Deliverables:

- Job seeker onboarding wizard.
- Staff task queues.
- Weekly job search review.
- Better mobile staff/user workflows.
- Global search.
- Notifications/reminders.
- Circle.so read integration if it supports a clear user workflow.

## Deployment Model

Early hosted rollout:

- Cloud Run direct service.
- `AUTH_MODE=session`.
- Turso/libSQL database.
- Secret Manager for credentials.
- Cloud Scheduler for sync/export.
- No external HTTPS load balancer.
- No IAP.

Cost controls:

- Artifact Registry cleanup policy.
- Budget alerts.
- Logging exclusions for low-value successful static requests.
- Max Cloud Run instances capped during beta.

## Open Decisions

- Whether RR staff access should be assignment-only or cohort-based for beta.
- Which user notes/check-in fields staff should see by default.
- Whether users can reject/archive RR-added jobs with feedback.
- Whether Sheets backup should be one workbook per org or per user.
- Which fields, if any, should push back to Sheets during the pilot.
- Whether Circle.so should remain reference-only or become an onboarding trigger.

## First Implementation Slice

Start with Phase 1 only.

Recommended first PR:

- Add organization/membership schema.
- Backfill one default organization.
- Add request-level role/tenant helpers.
- Scope one low-risk entity end-to-end.
- Add tests proving cross-user isolation.

Do not build the staff dashboard until the access model is enforced in the backend.
